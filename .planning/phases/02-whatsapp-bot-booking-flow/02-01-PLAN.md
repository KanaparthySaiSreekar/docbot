  <name>Task 1: WhatsApp Cloud API client with retry</name>
  <files>src/docbot/whatsapp_client.py, tests/test_whatsapp_client.py, pyproject.toml</files>
  <action>
Create `src/docbot/whatsapp_client.py` with the WhatsApp Cloud API client.

**WhatsApp Cloud API base URL:** `https://graph.facebook.com/{api_version}/{phone_number_id}/messages`
- `api_version` and `phone_number_id` from `settings.whatsapp`
- Authorization: `Bearer {access_token}` header

**Functions to implement:**

1. `async def send_text(to: str, text: str) -> dict | None` - Send plain text message.
   Payload: `{"messaging_product": "whatsapp", "to": to, "type": "text", "text": {"body": text}}`

2. `async def send_buttons(to: str, body_text: str, buttons: list[dict]) -> dict | None` - Send interactive reply button message (max 3 buttons).
   Each button: `{"type": "reply", "reply": {"id": button_id, "title": button_title}}`
   Payload: `{"messaging_product": "whatsapp", "to": to, "type": "interactive", "interactive": {"type": "button", "body": {"text": body_text}, "action": {"buttons": buttons_formatted}}}`

3. `async def send_list(to: str, body_text: str, button_text: str, sections: list[dict]) -> dict | None` - Send interactive list message (for time slot selection where >3 options exist).
   Payload: `{"messaging_product": "whatsapp", "to": to, "type": "interactive", "interactive": {"type": "list", "body": {"text": body_text}, "action": {"button": button_text, "sections": sections}}}`

4. `async def _send_message(payload: dict) -> dict | None` - Internal helper that actually sends the HTTP POST with retry.
   - Use `httpx.AsyncClient` (already a dependency) for HTTP calls.
   - Retry on 5xx or network errors with exponential backoff: delays of 1s, 2s, 4s (3 attempts max).
   - Log each retry attempt at WARNING level.
   - On final failure, log at ERROR level with alert category "whatsapp_send_failure" using `alerts.log_alert()`.
   - Return response JSON on success, None on complete failure (FAIL-02: failures do not raise exceptions).
   - FAIL-01: Retry with backoff. FAIL-02: Log but don't block.

**Tests in `tests/test_whatsapp_client.py`:**
- Test send_text constructs correct payload (mock httpx).
- Test send_buttons constructs correct interactive payload.
- Test send_list constructs correct list payload.
- Test retry on 500 response (mock returns 500, then 200).
- Test final failure returns None after max retries.
- Use `unittest.mock.patch` or `pytest-httpx` (prefer unittest.mock.AsyncMock to avoid adding dependencies).

**pyproject.toml:** No new dependencies needed (httpx already present).

<task type="auto">
  <name>Task 2: WhatsApp webhook endpoint with verification and message parsing</name>
  <files>src/docbot/webhook.py, tests/test_webhook.py, src/docbot/main.py</files>
  <action>
Create `src/docbot/webhook.py` as a FastAPI APIRouter.

**Endpoints:**

1. `GET /webhook/whatsapp` - Webhook verification (Meta sends this during setup).
   - Query params: `hub.mode`, `hub.verify_token`, `hub.challenge`
   - If `hub.mode == "subscribe"` and `hub.verify_token` matches `settings.whatsapp.verify_token`, return `int(hub.challenge)` as plain text (Response with media_type="text/plain").
   - Otherwise return 403 Forbidden.

2. `POST /webhook/whatsapp` - Receive incoming messages.
   - Parse the webhook payload to extract: sender phone number (`from`), message type, message content.
   - WhatsApp webhook structure: `body.entry[0].changes[0].value.messages[0]`
   - Extract `message_id` (wamid) for idempotency deduplication.
   - Check idempotency: if message already processed, return 200 OK immediately.
   - For message types:
     - `"text"`: Extract `message.text.body`
     - `"interactive"`: Check `message.interactive.type`:
       - `"button_reply"`: Extract `message.interactive.button_reply.id` and `.title`
       - `"list_reply"`: Extract `message.interactive.list_reply.id` and `.title`
   - Create a parsed message dict: `{"from": phone, "type": msg_type, "message_id": wamid, "button_id": id_or_none, "button_title": title_or_none, "text": text_or_none, "timestamp": timestamp}`
   - For now, log the parsed message and return 200 OK. The actual bot handler will be wired in Plan 02-04.
   - Always return 200 to Meta (even on errors) to prevent webhook retry storms. Log errors internally.
   - Record the message_id via idempotency.record_event() after processing.

**Register router in main.py:**
Add `from docbot import webhook` and `app.include_router(webhook.router)` in main.py, similar to how auth.router is included.

**Tests in `tests/test_webhook.py`:**
- Test GET verification with correct token returns challenge.
- Test GET verification with wrong token returns 403.
- Test POST with text message extracts sender and text.
- Test POST with button reply extracts button_id.
- Test POST with list reply extracts list selection.
- Test POST with duplicate message_id (idempotency) returns 200 without reprocessing.
- Test POST with malformed body returns 200 (no crash).
- Use FastAPI TestClient from conftest.py.
